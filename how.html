Argus 是怎么工作的

把整套系统讲清楚:一个笨管道、三个 agent、一个碰不到的裁判。然后用两个真实例子走一遍——它怎么写一篇论文,怎么在 nanochat 上一步步发明。 想要逐条穷举的机制清单(每道门、每个阈值)→ 完整机制参考。

一句话模型

Argus = 一个故意做笨的"流水线管道" + 三个会吵架的 AI agent + 一个被冻结、agent 永远碰不到的"裁判"。 7×24 不停跑。你只写一段"我要什么"的规格,它自己定题、写代码、跑实验、写论文,跑完告诉你结果——包括它做不到什么。

关键一句:管道不聪明,聪明的是 agent;胜负怎么算,永远由外部冻结的裁判说了算。这是整套系统的命根——因为裁判在 agent 够不到的地方,agent 就没法"作弊刷分",可以被逼到极度激进却依然诚实。

三个角色(会互相对抗)

PPlanner 规划

定方向、把大目标拆成一个个任务、判断"这个项目做完了没"。像项目经理。

EEngineer 工程

真正干活的:写代码、跑实验、profile 找瓶颈、改内核、写 LaTeX。像研究员/工程师。

RReviewer 评审

专门来挑刺的对手:找漏洞、要求更大胆、判这步过不过。它是"完成"的唯一裁定者。

三个角色各跑各的大模型,谁也不替谁背书——Engineer 说"我做完了",Reviewer 不信、来挑刺。这种对抗就是质量的来源。

一轮是怎么跑的

# 管道每一轮(round)做的事: Planner ──► 挑一个任务 给 Engineer Engineer ──► 写代码 / 跑实验 ──► 产物(代码、结果、图) Reviewer ──► 审这步 ──► done / continue(带具体修改要求) / blocked 管道 ──► 记状态、过门禁(页数够没?预算超没?)、需要就回滚 └─► 反复，直到 Reviewer 判"整个项目完成"
注意:管道只做"搬运状态 + 检查形式(能不能编译、超没超预算)+ 回滚"。它从不判断"这个想法好不好"——那是 agent 的事。

案例 A:写一篇论文(8 个阶段)

用真实的 MMR-Trap 论文走一遍。Planner 把目标拆成 8 个阶段,每个阶段 Engineer 干活、Reviewer 把关,过了才进下一步:

01

research 定题

挖文献、找缺口、定贡献句(X/Y/Z/W:我做了啥、缺口在哪、证据是啥、所以呢)。

→ 产物:RESEARCH_BRIEF.md、候选想法
02

plan 实验方案

设计要跑哪些实验、对照组、用哪些模型/数据。Reviewer 审方案能不能证明贡献。

→ 产物:EXPERIMENT_PLAN.md
03

benchmark 建评测

准备真实评测数据、验证 gold answer、做好可复现的评分器(这就是冻结裁判)。

→ 产物:bench/ 数据 + verifier.py
04

run 跑实验

在真模型上跑(MMR-Trap 跑了 13 个 VLM),每个 cell 多 seed,记原始结果。

→ 产物:experiments/ 结果、summary.tsv
05

analysis 出结果

聚合成表、画图、和已发表 baseline 对比(对不上要解释)。

→ 产物:图、结果表、RESULTS.md
06

draft 写 LaTeX

按会议模板(EMNLP/AAAI)写正文、嵌图、加引用。

→ 产物:paper/main.tex
07

review 自审改稿

Reviewer 当"对手审稿人"挑学术语言、逻辑漏洞、claim 有没有证据,Engineer 改。

→ 产物:修订后的稿子
08

submission 出 PDF

过完所有结构门(页数、引用数、无 overfull、图都在),编译出最终 PDF。

→ 产物:paper/main.pdf ✅
诚实边界:这 8 步保证产出一篇形式完整、能编译的论文(已出 2 篇)。但门禁测的是形式,不是贡献真不真——那个还得人和审稿人判。

案例 B:nanochat 一次发明迭代

任务:5 分钟内把 nanochat 的 val BPB 压低。这是优化型任务,不写论文,循环是"剖析→发明→测量"。看一次真实迭代(最后赢的那个 q/k 接缝就是这么来的):

①

PROFILE 先剖析

Engineer 先 profile,找出时间花在哪——发现 q/k 的 rotary+RMSNorm 是个热点(1.078ms,占 4.27%)。不许上来就瞎调参。

②

INVENT 发明一个新机制

针对那个热点,自己写一个融合内核:把 rotary 和 RMSNorm 合成一个 Triton 前后向,省掉中间存取。禁止读答案(读已发表最优解=判负)。

③

MEASURE 冻结裁判测

在冻结的评分器上跑 10 个 seed,取均值:0.98738,压过公平地板 0.98788。

④

判定 + 归档

Reviewer 判:单机制、过了地板 → PASS-A(真发明)。没过就写 PASS-B(诚实负结果),归档失败原因。

⑤

成熟期 + 不毙太早

大胆的结构改动第一轮常常更差;给它 2-4 轮"养"再判死。连续没进展才换一条全新方向(basin-hop)。

为什么可信(它和别人最大的不同)

  • 裁判冻结、外置:评分器、计时、验证集都锁死在 agent 够不到的地方,它没法刷自己不拥有的分。
  • 防作弊是硬门:"别读答案"是会判负的强制规则 + 隔离作弊配方,不是一句客气话。
  • 硬件诚实:H100 和 B200 不可比,跨硬件比较强制标注,绝不混着报。
  • 失败是一等公民:测出来没赢(PASS-B)照样写清楚归档,不藏。
一句话:别人比"涨了多少",Argus 比"涨的这点经得起审计"。代价是头条数字更小;好处是没人能用"夸大"打它。
